# Address Issues — Generate & Execute Fix Prompts

> **Purpose:** Read `issues.md`, decompose every finding into self-contained fix prompts, generate a master loop and state tracker, then optionally begin execution.
>
> **Input:** `issues.md` at the repo root (generated by `prompts/standard/repo-eval.md`)
> **Output:** `prompts/arch/r{NNN}/` containing numbered `FIX-NN.md` prompts, `MASTER.md`, and `STATE.md`

---

## Prerequisites

- `issues.md` exists at the repo root and contains at least one finding.
- If `issues.md` is empty or missing, STOP and tell the user to run the repo-eval prompt first.

---

## Step 1 — Determine the Revision Number

Scan `prompts/arch/` for existing `r###` directories. The next revision is `max + 1`, zero-padded to 3 digits.

```
Example: r001 exists → create r002
         r001, r002 exist → create r003
         nothing exists → create r001
```

Store this as `R_DIR` for the rest of the prompt (e.g., `prompts/arch/r002`).

---

## Step 2 — Read and Parse issues.md

Read `issues.md` in full. Extract every finding — each `### N.N {title}` block is one finding.

For each finding, capture:

| Field | Source |
|-------|--------|
| **ID** | The section number (e.g., `1.3`, `3.7`) |
| **Title** | The heading text |
| **Severity** | 🔴 Critical / 🟠 High / 🟡 Medium / 🟢 Low |
| **Category** | Chat Bleed / CLI Activity Monitor / Race Condition / Error Handling / Memory / Security / Architecture / Code Smell |
| **Location** | File path(s) and line ranges |
| **Description** | What the issue is |
| **Evidence** | The specific code cited |
| **Impact** | What could go wrong |
| **Fix** | The recommended fix from the evaluation |

Also capture any **Coverage Matrix** tables (CLI Activity Monitor section) — these inform which surfaces need fixes.

If the "Positive Observations" section exists, note the good patterns to preserve.

---

## Step 3 — Group and Order Findings

### 3a. Group by Affinity

Findings that touch the same file or subsystem should be combined into a single prompt when:

- They affect the same file and can be fixed together without the prompt exceeding ~200 lines of instructions
- Fixing one inherently fixes or changes the other
- They share the same root cause

**Do NOT combine** findings that:

- Span different architectural layers (e.g., a renderer store fix + an infrastructure fix)
- Have different severity levels where the lower one might be deferred
- Would make the prompt too complex to verify in isolation

### 3b. Assign Prompt Numbers

Number prompts sequentially: `FIX-01`, `FIX-02`, etc.

### 3c. Determine Dependencies

A prompt depends on another when:

- It modifies a file that the other prompt creates or heavily restructures
- It relies on a type, interface, or utility introduced by the other prompt
- The fix approach changes depending on whether the other fix was applied first

**Default:** Most fixes are independent. Only add dependencies when there's a real data flow reason.

### 3d. Determine Execution Order

Sort by:

1. **Critical (🔴) findings first** — these are active bugs
2. **High (🟠) next** — likely user-visible
3. **Medium (🟡) after** — architectural improvements
4. **Low (🟢) last** — nice-to-haves
5. Within the same severity, independent prompts before dependent ones

### Prompt Quality Rules

- **Read before writing.** Every implementation step must start with "Read `file.ts`" before modifying it.
- **Be surgical.** Don't rewrite entire files. Cite the specific method, the specific line range, the specific change.
- **Show the pattern.** For non-obvious fixes, include a before/after code snippet.
- **One concern per prompt.** Each prompt fixes one logical issue (even if it touches multiple files). Don't bundle unrelated fixes.
- **Verification is mandatory.** Every prompt must have at least: (a) `npx tsc --noEmit` passes, (b) one behavioral verification (grep, line count, or test scenario).
- **No TODOs.** The fix prompt must contain enough information for an agent to execute it without asking questions.

## Step 6 — Generate MASTER.md

Create `{R_DIR}/MASTER.md`:

````markdown
# Issue Fixes — Master Loop (r{NNN})

> Run this prompt to execute issue fixes. It reads `STATE.md`, picks the next prompt, executes it, updates state, and loops until all prompts are done.

---

## Context

These prompts address findings from `issues.md`, generated by the repo evaluation prompt. Each prompt is a self-contained fix with clear inputs, outputs, and verification steps.

**Source evaluation:** `issues.md` ({date})
**Total prompts:** {count}

---

## Instructions

You are executing issue fixes for Novel Engine. This is a set of {N} encapsulated prompts that address specific code issues found during evaluation.

**Your job:** Loop through the prompts in priority order until all are `done`.

---

## Protocol

### On each iteration:

1. **Read state.** Read `{R_DIR}/STATE.md` to see what's done, what's pending, and any handoff notes from previous runs.

2. **Pick the next prompt.** Select the first `pending` prompt whose dependencies are satisfied. Prefer higher severity prompts when multiple are eligible.

   Dependencies:
   {List each prompt's dependencies, same format as r001/MASTER.md}

3. **Read the prompt.** Read `{R_DIR}/FIX-NN.md` in full.

4. **Read affected files.** Before modifying any file, read it completely. Check for changes from previously completed prompts in this revision.

5. **Execute it.** Follow the prompt's implementation steps precisely. Write complete, production-ready code. Respect all architecture rules (layer boundaries, DI, no business logic in IPC handlers).

6. **Verify.** Run every verification step listed in the prompt. If verification fails, fix the issue before proceeding.

7. **Update state.** Edit `{R_DIR}/STATE.md`:
   - Set the prompt's status to `done`
   - Set the Completed date to today
   - Add any notes about decisions made or complications encountered
   - Update "Last completed prompt" and "Observations" in Handoff Notes

8. **Update documentation.** Follow the AGENTS.md documentation protocol:
   - Append a CHANGELOG.md entry for this prompt's changes
   - Update affected architecture docs (DOMAIN.md, INFRASTRUCTURE.md, APPLICATION.md, IPC.md, RENDERER.md, ARCHITECTURE.md)

9. **Loop.** Go back to step 1. If all prompts are `done`, report completion.

## Stopping Conditions

- **All done:** All {N} prompts are `done` in STATE.md. Report final summary.
- **Blocked:** A prompt fails verification and you cannot fix it. Set status to `blocked` with notes. Skip to next eligible prompt.
- **Context limit:** If you're running low on context, update STATE.md with current progress and stop. The next run will pick up where you left off.
- **Deferred:** If a fix requires changes outside the app's control (e.g., upstream CLI behavior), set status to `deferred` with notes.

---

## Final Report

When all prompts are done, produce a summary:

1. Total prompts completed
2. Total prompts blocked/deferred
3. Total files created
4. Total files modified
5. Findings addressed by severity (🔴: N, 🟠: N, 🟡: N, 🟢: N)
6. Any remaining issues or follow-up work
7. Architectural impact summary
````

## Edge Cases

- **Empty issues.md:** Stop immediately. Tell the user to run `prompts/standard/repo-eval.md` first.
- **Only 🟢 Low findings:** Generate prompts anyway but note in MASTER.md that all are optional improvements.
- **Single finding:** Still generate the full MASTER.md + STATE.md structure. Consistency matters.
- **Findings that contradict each other:** Pick the architecturally cleaner fix. Note the contradiction in the prompt.
- **Findings already fixed:** If reading the code shows a finding is no longer valid (fixed since the eval), add it to "Findings Not Addressed" in STATE.md with reason "Already resolved."
- **Findings requiring new dependencies:** Note in the prompt's implementation steps. The agent executing the fix will install them.
- **More than 20 findings:** Consider splitting into two revisions (r002 for Critical/High, r003 for Medium/Low). But prefer one revision unless the prompt count exceeds 25.

---

## What This Prompt Does NOT Do

- It does **not** execute the fixes. It only generates the prompts.
- It does **not** modify any source code.
- It does **not** update CHANGELOG.md or architecture docs (that happens during execution).
- It does **not** re-evaluate the codebase. It trusts `issues.md` as the source of truth.
